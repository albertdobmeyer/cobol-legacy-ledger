### Integration Points

1. **COBOL → Python Bridge**:
   - Bridge executes COBOL binary as subprocess with cwd=banks/{NODE}/
   - COBOL writes pipe-delimited output to stdout
   - Bridge parses output, maps to SQLite schema
   - Error codes passed through (00, 01, 02, 03, 04, 99)

2. **COBOL → File System**:
   - COBOL programs open ACCOUNTS.DAT, TRANSACT.DAT with LINE SEQUENTIAL
   - Files are fixed-width text (ACCTREC 70 bytes, TRANSREC 103 bytes)
   - TRANSACT.cob appends to TRANSACT.DAT (grows with each transaction)
   - ACCOUNTS.cob reads/writes ACCOUNTS.DAT (rewrites full file on UPDATE)

3. **Python → Integrity Chain**:
   - After each transaction, bridge calls chain.append(tx_id, account_id, ...)
   - Chain generates SHA-256(contents + prev_hash) and HMAC-SHA256(hash, secret_key)
   - Entry stored in chain_entries table

4. **Python → SQLite Schema**:
   - accounts: id, name, account_type, balance, status, opened_at, last_activity
   - transactions: id, account_id, type, amount, date, time_str, description, status, batch_id, balance_after, created_at
   - chain_entries: chain_index, tx_id, account_id, tx_type, amount, timestamp, description, status, tx_hash, prev_hash, signature
   - api_keys: key_id, key_hash, role, label, created_at

5. **Scripts → Node Setup**:
   - build.sh: Compiles COBOL, creates cobol/bin/
   - seed.sh: Creates banks/{NODE}/ directories, writes ACCOUNTS.DAT, TRANSACT.DAT, BATCH-INPUT.DAT, generates .server_key for each node
   - setup.sh: Creates python/venv, installs requirements
   - demo.sh: Runs Phase 1 end-to-end (init-db → seed-demo → run-batch → verify-chain)

## Design Decisions

### Decision 1: Per-Node Transaction ID Format

**Format**: `TRX-{node}-{6-digit seq}` (exactly 12 characters)
- Examples: `TRX-A-000001`, `TRX-B-000015`, `TRX-X-000003`
- Node codes: A=BANK_A, B=BANK_B, C=BANK_C, D=BANK_D, E=BANK_E, X=CLEARING

**Rationale**:
- Sequential IDs reflect batch processing order (auditable)
- Timestamps are in TRANS-DATE (separate concern)
- Standard COBOL pattern: file IS the state (read max seq from TRANSACT.DAT, continue from there)
- Fits 12-char TRANS-ID field exactly

**Alternatives Considered**:
- A1: Timestamped (TRX-20240101101530) — too long for 12-char field
- A2: Random UUID fragment — not sortable, no processing order visibility

**Decision**: Per-node sequential (chosen format)

---

### Decision 2: Per-Node HMAC Secret Keys

**Format**: `banks/{NODE}/.server_key` (separate file per node)
- Generated by seed.sh: `secrets.token_hex(32)` → 64-char hex string
- Loaded by COBOLBridge(node="BANK_A") from `banks/BANK_A/.server_key`
- Auto-generated if missing (on first run)
- Permissions: 0o600 (owner read-write only)

**Rationale**:
- Each node signs its own chain (self-certifying)
- Tampering with BANK_A's ledger requires BANK_A's key
- Phase 2: Clearing house verifies all banks using their keys
- Simulates trust boundaries (separation of key storage)

**Alternatives Considered**:
- A1: Shared key for all nodes — allows re-signing (verification is theater)
- A2: Asymmetric crypto (private key per node, public key for verification) — too complex for Phase 1 demo
- A3: Clearing house signs all chains — loses node independence

**Decision**: Per-node keys, self-certifying chains (chosen)

---

### Decision 3: Smoke Test Before ACCOUNTS.cob Implementation

**Order**: Run SMOKETEST.cob first, verify output format, then implement ACCOUNTS.cob

**Rationale**:
- Validates compiler is working correctly
- Confirms exact balance format (PIC S9(10)V99 representation, no literal decimal)
- Python bridge parser must understand this format
- Prevents mismatch between COBOL output and Python parsing expectations

**Alternatives Considered**:
- A1: Smoke test as part of ACCOUNTS task — loses early validation
- A2: Diagnostic mode only — too late to catch compiler issues

**Decision**: Smoke test FIRST, then implement (chosen)

---

### Decision 4: FILE STATUS on All File Operations

**Implementation**: Every SELECT statement includes `FILE STATUS IS WS-FILE-STATUS`

```cobol
SELECT ACCOUNTS-FILE
    ASSIGN TO ACCOUNTS-DAT
    ORGANIZATION IS LINE SEQUENTIAL
    FILE STATUS IS WS-FILE-STATUS.
```

**Rationale**:
- Distinguish file errors (99) from business logic errors (01-04)
- Bridge can parse status codes and handle gracefully
- Missing ACCOUNTS.DAT triggers error code 30, not hard crash

**Alternatives Considered**:
- A1: FILE STATUS only on critical operations — inconsistent error handling
- A2: No FILE STATUS, rely on exit codes — bridge can't distinguish error types

**Decision**: FILE STATUS on all SELECT statements (chosen)

---

### Decision 5: Compliance CTR Detection in COBOL, Not Python

**Implementation**: TRANSACT.cob detects deposits/transfers > $9,500, outputs compliance warning

```
COMPLIANCE|CTR-FLAG|{TRANS-ID}|{AMOUNT}
```

**Rationale**:
- Business logic lives in COBOL (source of truth)
- Python bridge relays warning, doesn't validate
- Detects structuring patterns (staying under $10k CTR threshold)
- Demo moment: Elena Petrov's $9,500 deposit triggers flag

**Alternatives Considered**:
- A1: Python bridge validates — violates COBOL immutability principle
- A2: No compliance checking — misses demo moment

**Decision**: COBOL detects, Python relays (chosen)

---

### Decision 6: Per-Node Database Files (Not Shared Ledger)

**Format**: `data/{node_lowercase}.db` (separate SQLite file per node)
- Examples: `data/bank_a.db`, `data/bank_b.db`, `data/clearing.db`

**Schema per node**:
- accounts (8 or 5 records depending on node)
- transactions (grows with each batch run)
- chain_entries (one per transaction, tamper-evident)
- api_keys (auth for this node's CLI)

**Rationale**:
- Financial systems are distributed (nodes operate independently)
- Per-node isolation enables Phase 2 cross-node verification
- No shared database = no cross-node table joins = cleaner architecture
- Mirrors production: each bank has its own ledger

**Alternatives Considered**:
- A1: Shared ledger.db with node column — enables Phase 2 joins, but violates distribution principle
- A2: In-memory only — no persistence

**Decision**: Per-node SQLite files (chosen)

---

### Decision 7: Path Resolution Strategy

**Binary path**: Resolved to absolute path in COBOLBridge.__init__()
```python
self.bin_dir = Path(bin_dir).resolve()  # Absolute: /full/path/to/cobol/bin
```

**Working directory**: Resolved for each subprocess call
```python
cwd = Path(f"banks/{self.node}").resolve()  # Absolute: /full/path/to/banks/BANK_A
```

**Rationale**:
- Prevents CWD-relative issues (if caller cd's somewhere, paths don't break)
- COBOL subprocess opens data files relative to cwd (banks/{NODE}/)
- Binary path must be absolute so subprocess finds it regardless of caller's CWD

**Alternatives Considered**:
- A1: Relative paths — breaks if caller cd's to different directory
- A2: Hardcoded absolute paths — not portable across machines

**Decision**: Resolve both binary and cwd to absolute (chosen)

---

### Decision 8: Seeding Modes (Mode A with COBOL, Mode B without)

**Mode A** (if cobc available):
1. seed.sh writes ACCOUNTS.DAT in fixed-width format
2. bridge.load_accounts_from_cobol() executes ACCOUNTS binary
3. COBOL outputs pipe-delimited records to stdout
4. Bridge parses output into dicts, writes to SQLite

**Mode B** (if cobc unavailable):
1. seed.sh writes ACCOUNTS.DAT in fixed-width format (Python writes text)
2. generate_demo_data() reads ACCOUNTS.DAT directly (Python file I/O)
3. Python parser reads fixed-width records manually (using ACCTREC.cpy layout)
4. Bridge writes parsed dicts to SQLite

**Rationale**:
- System works on machines without GnuCOBOL
- Both modes produce identical SQLite state
- Demo doesn't require COBOL (fallback is built-in)

**Alternatives Considered**:
- A1: COBOL required — fails on machines without cobc
- A2: Python only — ignores COBOL, misses the point of Phase 1

**Decision**: Both modes, auto-detect available (chosen)
